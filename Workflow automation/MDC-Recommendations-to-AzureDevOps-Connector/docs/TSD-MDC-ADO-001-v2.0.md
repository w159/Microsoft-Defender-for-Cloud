## 0. Change Log — What's New in v2

| Change | Rationale |
|---|---|
| Introduced **Azure Function App** as the enrichment and orchestration engine | The Logic App alone cannot perform multi-source enrichment (Azure Resource Graph, MDC attack paths, vulnerability subassessments, governance assignments) without becoming unmaintainable. |
| Logic App reduced to **trigger + dispatch** role | MDC Workflow Automation only supports Logic App as a target. The Logic App now exists solely to satisfy that contract and forward the payload to the Function. |
| **Language: Python 3.12 (v2 programming model)** | Selected for richer enrichment/AI ecosystem alignment, faster iteration, and lighter footprint. Python 3.12 is GA-supported on Azure Functions v4. |
| Work Items now include a rich **"Triage Briefing"** section auto-generated by the Function | Engineers receive a decision-ready ticket instead of raw recommendation text, dramatically reducing time-to-triage. |
| Added enrichment data sources: **Attack Paths, other open recs, vulnerabilities (CVEs), suggested owner, resource criticality, exposure** | Provides the engineer with full context at triage time, including blast radius and prioritization signals. |
| Adopted **Durable Functions** for the orchestration layer inside the Function App | Built-in retries, parallel activity fan-out, status queryability, testability. |
| Added v2 permissions: `Reader` and `Security Reader` for the Function MI | Required for ARG and MDC queries. |
| **ADO authentication via the Function's Managed Identity (no PAT)** | Eliminates the stored secret, the Key Vault dependency for ADO, and the PAT rotation burden. Azure DevOps supports adding a Managed Identity / service principal as an organization member with scoped permissions. |
| **Hosting on the Flex Consumption plan** | GA plan with faster cold starts (always-ready instances), per-instance concurrency, and VNet integration support. Replaces the classic Consumption/Premium choice. |
| Manual closure model from v1.1 is **retained** (MDC constraint unchanged) | See §2.4. Reconciliation Logic App is still parked as a v3 enhancement. |

## 1. Executive Summary

This document specifies v2 of the automated connector that creates and lifecycle-manages Azure DevOps (ADO) Work Items from Microsoft Defender for Cloud (MDC) Security Recommendations.

v2 evolves the v1 design by introducing an **Azure Function App** as an enrichment engine sitting behind the Logic App. The Logic App continues to satisfy MDC Workflow Automation's contract (LA is the only supported trigger target), but immediately forwards the recommendation payload to the Function. The Function performs multi-source enrichment — Attack Paths, related recommendations, vulnerability findings, suggested owner, resource criticality, exposure signals — and then composes a **Triage Briefing** that is posted into the ADO Work Item.

The design intentionally avoids ITSM-grade tooling (e.g., ServiceNow) and instead implements a **"Service Request-alike"** pattern in ADO Boards, now substantially enriched.

## 2. Background and Business Context

### 2.1 Problem Statement
v1 delivered the core trigger → create Work Item flow but produced "thin" Work Items that contained only the raw recommendation text and resource ID. Engineers triaging these items had to manually pivot into the MDC portal, Azure Resource Graph, and other tools to determine:

- Is this resource on an attack path?
- How many other recommendations does it have?
- Does it have known vulnerabilities (CVEs)?
- Who owns it?
- Is it business-critical?

This investigative overhead per ticket undermined the value of the automation. v2 closes that gap by performing the enrichment automatically and embedding it directly into the Work Item.

### 2.2 Goals (v2)
All v1 goals, plus:
- Reduce per-ticket triage time by providing a decision-ready **Triage Briefing**.
- Surface **blast-radius signals** (attack paths, other recs, criticality) at creation time.
- Surface **technical evidence** (CVE list, exposure) inline in the Work Item.
- Provide an extensible enrichment pipeline for future signals (Teams notifications, AI summarization, etc.).

### 2.3 Non-Goals (v2)
- Full ITSM functionality (CMDB, change advisory boards, approvals workflow).
- Bi-directional sync of arbitrary Work Item edits back to MDC.
- Multi-tenant fan-out across multiple ADO organizations.
- **Automated closure of Work Items upon remediation** (engineers close manually — see §2.4).
- Centralized observability via Log Analytics (deferred to v3).
- Auto-reconciliation Logic App (deferred to v3).

### 2.4 Platform Constraints (MDC) — Unchanged from v1.1

1. **No remediation event / no history retention** — when a Security Recommendation transitions to remediated, the assessment record is cleared from MDC. There is no `Healthy` event to subscribe to.
2. **No back-reference storage** — MDC does not expose a *dedicated* field to persist the ADO Work Item ID against an assessment.
3. **Consequence** — automatic Work Item closure cannot be implemented. Engineers close manually after verifying remediation. A reconciliation Logic App (scheduled ARG diff) remains parked as a v3 option (see §13).
4. **v2.1 partial mitigation of #2** — the optional **recommendation assignment write-back** (§4.7) stamps the Work Item id into the assessment's *governance assignment* `owner` field (the only writable back-reference MDC exposes) and flips the recommendation to **Assigned**. This is a one-time write per material change, not bi-directional edit sync (still a non-goal, §2.3), and does **not** enable automatic closure.

### 2.5 Prerequisites & Assumptions (v2)

1. **Defender CSPM (DCSPM) is mandatory** on all in-scope subscriptions. It is required for the core enrichment signals — Attack Paths and the `securityresources` cloud-security graph used for blast-radius context, including critical-asset insights surfaced on attack-path targets (§4.3 #1–#3, #7). The connector is not expected to deliver its triage value without DCSPM enabled.
2. **CVE / vulnerability enrichment is best-effort** and additionally depends on **Defender for Servers** and/or **Defender for Containers** being enabled on the relevant resources (§4.3 #4). Where these plans are off, the CVE section degrades gracefully (per §5.2.3) and the Work Item is still created.
3. **Single tenant** for v2 (multi-tenant fan-out is out of scope — §2.3).
4. **The Azure DevOps organization is backed by the same Entra tenant**, enabling the Function's Managed Identity to be added as an org member for authentication (§6.1).

## 4. Functional Design

### 4.1 Use Cases

| ID | Use Case | Trigger |
|---|---|---|
| UC-01 | Create new enriched Work Item from new MDC recommendation | Assessment with status `Unhealthy` (new) |
| UC-02 | Update existing Work Item on recommendation refresh | Assessment re-fired, WI exists |
| UC-03 | Skip noise (filter low severity, irrelevant scope) | Filter logic in LA |
| UC-04 | Degrade gracefully on partial enrichment failure | One or more ARG/MDC calls fail |
| UC-05 | Surface error/failure to ops team | Run failure in LA or Function |

> **Out of scope for v2:** automated Work Item closure on remediation (MDC constraint per §2.4). Engineers close Work Items manually.

### 4.2 Filtering Rules (v2)

Filtering is applied in **two tiers**, with the coarsest filter pushed as far upstream as possible:

1. **MDC Workflow Automation rule (primary filter).** The automation rule's own conditions are scoped so that only relevant assessments ever fire the Logic App — primarily by **severity** (`High`, `Medium`) and **assessment name/type**. Filtering here means out-of-scope events never incur Logic App or Function execution at all (lowest cost, lowest noise).
2. **Logic App (secondary / fine-grained filter).** Conditions the Workflow Automation rule cannot express (e.g., subscription allowlists, tag checks) are evaluated in the Logic App before the Function is invoked.

Net effect — process the recommendation **only if** all of the following hold:
- `properties.metadata.severity` ∈ {`High`, `Medium`}  *(enforced primarily at the Workflow Automation rule)*
- `properties.resourceDetails.id` matches the in-scope subscription list  *(Logic App)*
- `properties.metadata.assessmentType` ∈ {`BuiltIn`, `CustomerManaged`}
- (Optional) Resource has tag `SecurityTracked = true`  *(Logic App)*

### 4.3 Enrichment Data Model

The Function gathers the following signals per recommendation. All are best-effort — partial failure does not prevent Work Item creation.

| # | Signal | Source | API / KQL |
|---|---|---|---|
| 1 | Other open recommendations on the affected resource | ARG `securityresources` | `where type == 'microsoft.security/assessments' and properties.resourceDetails.id =~ '<rid>' and properties.status.code == 'Unhealthy'` |
| 2 | Of those, how many are assigned vs unassigned | MDC Governance Assignments | `securityresources` type=`microsoft.security/assessments/governanceassignments` joined with #1 |
| 3 | Attack paths the resource participates in | MDC CSPM (DCSPM) | `securityresources` where `type == 'microsoft.security/attackpaths'`, matched to the resource via `properties.targetEntityInternalId` / `entryPointEntityInternalId`. Returns `displayName`, `riskFactors`, MITRE tactics/techniques, `attackPathSteps`, `manualRemediationSteps`. **Verified live 2026-06-23.** |
| 4 | Vulnerabilities / CVEs (VMs, containers, images) | Defender Vulnerability Mgmt subassessments | `securityresources` type=`microsoft.security/assessments/subassessments` joined to vuln assessment IDs |
| 5 | Suggested owner | Resource tags → optional Graph lookup | ARG `resources` where `id =~ '<rid>'` → `tags.Owner` or `tags.SecurityContact` |
| 6 | Resource criticality | Resource tags (primary) + attack-path insights | **No standalone ARG criticality type exists.** Critical-asset classification now lives in Defender XDR / Exposure Management and is surfaced via ARG only as an *insight on attack-path targets* (see #3). Primary source is `tags.Criticality`; enriched opportunistically from attack-path data when present. Abstract behind `IResourceCriticalityProvider`. |
| 7 | Internet exposure | Attack path metadata | Derived from #3 — `properties.riskFactors` includes `Internet exposure` when applicable. **Verified live 2026-06-23.** |
| 8 | Resource type, location, subscription, RG | Resource ID parse + ARG `resources` | Direct |
| 9 | Compliance standards affected | MDC assessment metadata | Already in trigger payload |
| 10 | Secure Score impact | MDC assessment metadata | Numeric points value from `properties.metadata` (not a Low/Med/High label); already in trigger payload |

**Implementation note:** signals 1, 2, 3, 4, 7, 8 are served by **Azure Resource Graph** (`securityresources` / `resources`) and should be combined into **one or two ARG batches** via the ARG REST `POST /providers/Microsoft.ResourceGraph/resources` endpoint to respect throttling limits. Signal 5 (owner) uses ARG `resources` plus an optional Microsoft Graph lookup. **ARG support verified 2026-06-23** against a live DCSPM-enabled subscription: a `microsoft.security/attackpaths` query returned a complete attack-path object (display name, risk factors, MITRE mapping, steps, remediation). **Exception — signal 6 (criticality):** there is no `microsoft.security/criticalassets` ARG type; criticality is tag-based with best-effort enrichment from attack-path target insights.

### 4.6 State Machine — Unchanged from v1.1

All state transitions are performed manually by engineers. The connector controls only Work Item creation and field updates. See §2.4 for rationale.

### 4.7 Recommendation Assignment Write-Back (v2.1, optional)

After a Work Item is created or materially updated, the Function optionally writes a **governance assignment** back to MDC. This flips the recommendation to **Assigned** in the MDC governance dashboard and persists the Work Item id as a back-reference (working around constraint §2.4 #2). It mirrors the pattern proven in the prior MDC→Jira connector.

- **Mechanism:** `PUT https://management.azure.com{assessmentId}/governanceAssignments/{key}?api-version=2025-05-04` with body `{"properties": {"owner": "ado-wi-<id>@<domain>", "remediationDueDate": "<SLA>", "isGracePeriod": true}}`, authenticated with the Function MI's ARM token.
- **Idempotency:** `{key}` is a deterministic GUID = `uuid5(namespace, "<assessmentKey>|<resourceId>")`, so repeated writes upsert the same assignment (no duplicates). Re-PUTting our own key is a clean update.
- **Existing assignments are preserved:** MDC permits only one governance assignment per assessment. If one already exists under a different key (e.g. a human owner), the write returns HTTP 409; the connector treats this as a clean skip (`already-assigned`) and never overwrites it.
- **Gating:** runs only on a material `created`/`updated` (never on a churn-suppressed `skip`, §5.2.5), and only when `MDC_WRITE_BACK_ENABLED` is set.
- **Owner format:** `ado-wi-<workItemId>@<MDC_ASSIGNMENT_OWNER_DOMAIN>` (default domain `ado.local`).
- **Feedback-loop guard:** enrichment (§4.3 #5) ignores governance owners with the `ado-wi-` prefix, so the connector never surfaces its own back-reference as a human owner or counts it as a human-assigned recommendation.
- **Best-effort (§5.2.3):** the Work Item already exists, so a write-back failure (e.g. the MI lacking `Security Admin`) is logged and never fails the orchestration.
- **Permission:** requires the Function MI to hold `Security Admin` on the in-scope subscription (§6.1).

## 5. Detailed Component Design

### 5.1 `la-mdc-ado-dispatcher` — Logic App

| Step | Action | Details |
|---|---|---|
| 1 | **Trigger** | `When a Microsoft Defender for Cloud Recommendation is created or triggered` |
| 2 | **Condition: In-scope?** | Filter by severity + scope (§4.2); if false → terminate(succeeded) |
| 3 | **HTTP action** | POST trigger payload to `https://fa-mdc-ado-enricher.azurewebsites.net/api/EnrichAndCreate` |
| 4 | **Auth** | Easy Auth (Entra) + Logic App Managed Identity (preferred); Function key fallback |
| 5 | **Condition: HTTP status code** | If non-2xx → notify Ops Teams channel |

The Logic App is intentionally minimal. All business logic lives in the Function.

### 5.2 `fa-mdc-ado-enricher` — Azure Function App

**Stack:** Python 3.12, Azure Functions v2 programming model, `azure-functions-durable` extension, and `azure-identity` for token acquisition. All enrichment is read through **Azure Resource Graph** via the ARG REST API called with `httpx` — `azure-mgmt-security` is intentionally **not** used because every MDC signal in §4.3 is served by the `securityresources` table, avoiding a redundant management-plane dependency. Owner resolution calls **Microsoft Graph directly over REST** with `httpx` rather than importing the heavyweight `msgraph-sdk`, materially reducing cold-start cost. Azure DevOps is called via REST with `httpx` (Entra token from the Function's Managed Identity — no PAT). Models via `pydantic` v2.

**Hosting:** **Flex Consumption** plan. Configure a small number of **always-ready instances** to keep cold starts off the orchestrator/HTTP-start path, with per-instance concurrency for activity fan-out. Flex Consumption also supports VNet integration if private networking to dependencies is later required.

**ADO authentication:** the Function's user-assigned Managed Identity is added as a member of the Azure DevOps organization with scoped Work Item permissions on the target Area Path. The Function acquires an Entra token for the Azure DevOps resource (application ID `499b84ac-1321-427f-aa17-267ca6975798`, scope `/.default`) via `azure-identity` and calls the ADO REST API directly — no PAT, no Key Vault secret.

#### 5.2.1 Function inventory

| Function | Type | Purpose |
|---|---|---|
| `EnrichAndCreate_HttpStart` | HTTP trigger | Entry point from Logic App; starts orchestration |
| `EnrichAndCreate_Orchestrator` | Durable orchestrator | Coordinates activities |
| `Activity_GatherEnrichment` | Durable activity | Issues parallel ARG queries |
| `Activity_ResolveOwner` | Durable activity | Tag → AAD lookup via Microsoft Graph |
| `Activity_BuildTriageBriefing` | Durable activity | Composes HTML briefing |
| `Activity_DedupeLookup` | Durable activity | WIQL query against ADO |
| `Activity_CreateOrUpdateWorkItem` | Durable activity | Final ADO API call |

#### 5.2.2 Orchestrator pseudocode

```python
import azure.durable_functions as df

@bp.orchestration_trigger(context_name="context")
def enrich_and_create_orchestrator(context: df.DurableOrchestrationContext):
    payload = context.get_input()

    # Fan out enrichment in parallel
    enrichment_task = context.call_activity("activity_gather_enrichment", payload)
    owner_task = context.call_activity("activity_resolve_owner", payload)
    enrichment, owner = yield context.task_all([enrichment_task, owner_task])

    briefing = yield context.call_activity(
        "activity_build_triage_briefing",
        {"payload": payload, "enrichment": enrichment, "owner": owner},
    )

    existing_wi_id = yield context.call_activity("activity_dedupe_lookup", payload)

    result = yield context.call_activity(
        "activity_create_or_update_work_item",
        {
            "payload": payload,
            "enrichment": enrichment,
            "owner": owner,
            "briefing": briefing,
            "existing_wi_id": existing_wi_id,
        },
    )
    return result
```

#### 5.2.3 Graceful degradation

Every enrichment call is wrapped in try/catch. On failure:
- The bundle field is set to `null` or empty.
- A diagnostic note is appended to the briefing.
- Application Insights logs the failure with full context.
- The Work Item is still created/updated.

#### 5.2.4 Idempotency

- Dedupe key remains `Custom.MDCAssessmentId + Custom.MDCResourceId`.
- The Durable Function instance ID is derived from the dedupe key (`sha256(assessment_id + resource_id).hexdigest()`) to prevent duplicate orchestrations if the Logic App retries.
- **Instance-ID reuse caveats:** a deterministic instance ID only dedupes while that instance still exists in the Durable task hub. Before starting a new orchestration, `HttpStart` must **check the existing instance status** (`get_status`) and:
  - if the instance is `Running`/`Pending` for the same key → treat the event as a duplicate, return `202`, and do **not** start a second orchestration;
  - if the instance is `Completed`/`Failed` → start a **new** orchestration with the same ID (legitimate re-fire / update path), which Durable permits once the prior instance is terminal.
- **History-purge interaction:** a purge policy on completed instances is needed to keep the task hub small, but purging erases the dedupe memory for that key. This is acceptable because the **authoritative** dedupe is the ADO WIQL lookup (`Activity_DedupeLookup`) keyed on `MDCAssessmentId + MDCResourceId`; the instance-ID mechanism only guards against rapid Logic App retries, not long-term uniqueness.

#### 5.2.5 Update-churn control (UC-02)

MDC refreshes assessments on every scan, so re-fired events are common. Blindly PATCHing the Work Item on every event would create a new revision each time — updating `LastSeen` alone would spam the WI history and pollute "recent activity" views. `Activity_CreateOrUpdateWorkItem` therefore:

- Computes a hash of the **material** fields only (severity, briefing body, attack-path count, CVE count / max CVSS, criticality, owner) and stores it in `Custom.MaterialHash`.
- Issues a PATCH **only when the material hash differs** from the stored value.
- When only `LastSeen` would change, it skips the PATCH or coalesces the timestamp update to at most once per configurable window (e.g., 24 h).
- Sets `FirstDetected` once on create and never rewrites it.

## 6. Security Design

### 6.1 Identity & Access

| Principal | Permissions | Purpose |
|---|---|---|
| Logic App User-Assigned MI | Function App `Invoker` (Easy Auth) | Invoke the Function via Entra (no secret) |
| Function App User-Assigned MI | `Reader` on in-scope subscriptions | ARG `resources` queries |
| Function App User-Assigned MI | `Security Reader` on in-scope subscriptions/MG | ARG `securityresources` queries (MDC assessments, attack paths, subassessments) |
| Function App User-Assigned MI | Microsoft Graph `User.Read.All` (application, optional) | Resolve tag owner emails to Entra identities |
| Function App User-Assigned MI (added as ADO org member) | ADO **Contributor** on the target Team / Area Path (Work Item create/update) | Create/update Work Items via the ADO REST API using an Entra token — replaces the PAT |
| Function App User-Assigned MI | `Security Admin` on in-scope subscriptions (**v2.1, optional**) | Write the recommendation governance assignment — set **Assigned** + Work Item back-reference (§4.7). Omit if write-back is disabled. |
| Workflow Automation | `Security Admin` (built-in) on subscription | Configure/read MDC assessments for the automation rule |

### 6.2 Secrets Management
- **No application secrets in v2.** ADO authentication uses the Function's Managed Identity (Entra token); inter-service auth (Logic App → Function) uses Easy Auth + Managed Identity. There is therefore no ADO PAT to store or rotate.
- **No Key Vault dependency** for the baseline flow. If a future signal requires a secret, introduce Key Vault at that point and grant `Key Vault Secrets User` to the Function MI.
- No secrets in code, Logic App definitions, or Function App settings.

### 6.3 Network
- Logic App: public endpoint, but inbound restricted by the MDC-owned trigger.
- Function App (Flex Consumption): public HTTPS endpoint protected by **Easy Auth (Entra) with the Logic App's Managed Identity** (preferred); Function key as fallback. Flex Consumption supports VNet integration if private ingress/egress is later required.
- Key Vault: not deployed in the v2 baseline (see §6.2). If added later, Flex Consumption VNet integration allows a Private Endpoint + firewall configuration.
- Outbound from Function: HTTPS to `management.azure.com`, `graph.microsoft.com`, `dev.azure.com`.

### 6.4 Data Classification
- Assessment data: **Confidential — Internal**.
- Owner email (PII): handled within Microsoft tenant only; not exported.
- CVE / vulnerability data: **Confidential — Internal**.

## 7. Observability and Operations

> **Note:** v2 does **not** include a centralized Log Analytics Workspace. Observability relies on:
> - **Logic App run history** (native, 90 days) for trigger and dispatch
> - **Application Insights** for the Function (required by the runtime, scoped to Function only)
> - **Azure Monitor metric alerts** on both resources
>
> A centralized SIEM-style LAW is deferred to v3.

### 7.1 Logging
- **Logic App run history** — primary diagnostic for trigger and dispatch failures.
- **Application Insights** (`appi-mdc-ado-enricher`, connection-string based — instrumentation keys are deprecated) — distributed traces across all activities, exceptions, custom metrics (`EnrichmentLatency`, `ArgQueryCount`, `AdoApiLatency`).
- **ADO audit log** — Work Item creation/update events.

### 7.2 Monitoring & Alerts

| Alert | Source | Condition | Severity |
|---|---|---|---|
| LA run failure | Logic App `RunsFailed` metric | > 3 failures in 15 min | Sev 2 |
| Function 5xx | App Insights `requests/failed` | > 5 in 15 min | Sev 2 |
| Function cold-start latency | App Insights `requests/duration` | P95 > 30s | Sev 3 |
| ARG throttled | App Insights custom event `ArgThrottled` | > 10 in 5 min | Sev 3 |
| ADO API throttled | App Insights custom event `AdoThrottled` | HTTP 429 from ADO | Sev 3 |
| Entra token failure (ADO/Graph) | Function error | Token acquisition / authorization failure | Sev 1 |
| Trigger gap | Logic App `RunsStarted` metric | No runs in 24h | Sev 3 |
| Enrichment degradation | App Insights custom event `EnrichmentPartial` | > 20% of runs in 1h | Sev 3 |

### 7.3 KPIs
- Recommendations processed / day
- Work Items created vs. updated
- End-to-end latency: MDC event → WI created (target P95 < 30 s)
- Enrichment completeness rate (target > 95%)
- Mean enrichment signals per WI

| Metric | Target | Notes |
|---|---|---|

## 9. Resilience and Error Handling

- **Durable Functions retry policy** on every activity: exponential, 4 retries, max 1 min interval.
- **Graceful degradation** on enrichment failures (per §5.2.3) — WI is always created with whatever data was obtained.
- **Idempotent orchestration** via deterministic instance IDs (per §5.2.4).
- **WIQL eventual consistency** mitigated via secondary check on `tag contains <assessmentId>`.
- **Dead-letter**: failed Durable orchestrations are inspectable via Application Insights and the Durable Functions HTTP management API for manual replay.
- **Logic App retry**: 4 retries on the HTTP call to the Function (exponential).

Acceptance criteria:
- 100% of in-scope new recs create exactly one WI.
- Re-fired recs update existing WI; never create duplicates.
- Out-of-scope recs are filtered (zero false positives in WI backlog).
- ≥ 95% of WIs include a fully populated Triage Briefing.
- WIs are still created when enrichment partially fails (degradation working).
- Manual closure workflow validated with security ops in UAT.

## 12. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Backlog flooding | High | High | Aggressive filtering in LA, severity gating |
| ARG throttling | Medium | Medium | Batch into 1–2 queries per invocation; exponential backoff |
| Function cold-start latency | Low | Medium | Flex Consumption always-ready instances keep cold starts off the critical path |
| Partial enrichment producing thin WIs | Medium | Low | Graceful degradation + clear diagnostic markers in briefing |
| DCSPM not enabled on a target subscription | Low | High | DCSPM is a hard prerequisite (§2.5); validate at onboarding. Without it, core blast-radius enrichment is unavailable |
| ADO throttling | Medium | Medium | Retries with backoff; managed-identity Entra token |
| WIQL race condition | Low | Medium | Secondary tag check |
| Owner resolution failure (Graph) | Medium | Low | Fall back to tag value as string; WI still created |
| MDC Critical Assets API is in preview | Medium | Low | Tag-based fallback; abstract behind `IResourceCriticalityProvider` |
| Multi-tenant ARG query auth complexity | Low | Medium | Out of scope for v2 (single tenant) |
